# 📐 江湖打工人项目管理方法论

**版本：** v1.0  
**创建日期：** 2026-04-15  
**适用范围：** 江湖打工人（武侠游戏）+ goskill + 未来项目

---

## 🎯 核心原则

### 1. 目标导向
- 所有任务必须对齐总体目标（Demo 5/15，EA 7/31）
- 不做"可能有用"的功能，只做"必须完成"的功能
- 每个 Sprint 必须有可演示的成果

### 2. 敏捷迭代
- 双周 Sprint 制（14 天一个周期）
- 每个 Sprint 结束必须有可运行、可测试的版本
- 快速反馈，及时调整

### 3. 透明可视
- 所有任务在看板上可见
- 每日更新进度
- 问题及时暴露，不隐瞒

### 4. 风险前置
- 高风险任务优先做
- 技术验证在 Sprint 1 完成
- 预留 20% 缓冲时间

## 🏃 Sprint 管理

### Sprint 周期

| 阶段 | 时间 | 说明 |
|------|------|------|
| Sprint 规划 | Sprint 开始前 1 天 | 确定本 Sprint 任务清单 |
| Sprint 执行 | 14 天 | 每日站会 + 任务执行 |
| Sprint 评审 | 最后 1 天 | 演示成果 + Kay 验收 |
| Sprint 回顾 | 评审后 | 总结经验 + 改进下一 Sprint |

## 📅 每日站会制度

### 站会时间
- **时间：** 每天结束时（建议 22:00-23:00）
- **时长：** 5-10 分钟
- **参与：** Kay + 小聋瞎

### 站会内容

**必须汇报的三件事：**

1. **今日完成** - 今天完成了什么任务（对照计划）
2. **遇到的问题** - 有什么阻碍需要解决
3. **明日计划** - 明天计划做什么

### 站会模板

```markdown
### 4/XX 站会

**日期：** 2026-04-XX
**Sprint：** Sprint X - 第 Y 周

**今日完成：**
- [x] 任务 1（计划中的 Sx-01）
- [x] 任务 2（计划中的 Sx-02）

**遇到的问题：**
- 问题 1：描述 + 影响 + 已尝试的解决方案
- 问题 2：如无，写"无"

**明日计划：**
- [ ] 任务 1（计划中的 Sx-03）
- [ ] 任务 2（计划中的 Sx-04）

**是否需要帮助：** 是/否（如是，说明需要什么帮助）

**工时统计：**
- 计划工时：Xh
- 实际工时：Xh
- 偏差：+Xh / -Xh
```

### 站会纪律

- ❌ 不允许跳过站会
- ❌ 不允许只写"无"或"正常"
- ✅ 必须对照计划文件检查任务完成情况
- ✅ 必须更新任务状态（🔲 → 🚧 → ✅）
- ✅ 问题必须当天暴露，不隐瞒

## 📊 进度跟踪

### 任务状态定义

| 状态 | 符号 | 说明 |
|------|------|------|
| Todo | 🔲 | 未开始 |
| In Progress | 🚧 | 进行中 |
| Blocked | 🛑 | 被阻塞（需说明原因） |
| Done | ✅ | 已完成（符合 DoD） |
| Cancelled | ❌ | 已取消（需说明原因） |

### 进度计算公式

```
Sprint 完成率 = 已完成任务数 / 总任务数 × 100%

工时偏差率 = (实际工时 - 计划工时) / 计划工时 × 100%

里程碑风险 = 当前日期 / Sprint 结束日期 - 完成率
  - >0.2: 高风险（红色）
  - 0.1-0.2: 中风险（黄色）
  - <0.1: 低风险（绿色）
```

每个 Sprint 必须绘制燃尽图：

## 🚨 风险管理

### 风险等级

| 等级 | 颜色 | 说明 | 响应时间 |
|------|------|------|----------|
| 高 | 🔴 | 可能导致项目失败 | 24 小时内 |
| 中 | 🟡 | 可能影响进度 | 3 天内 |
| 低 | 🟢 | 轻微影响 | 本周内 |

### 风险应对策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 规避 | 改变计划避免风险 | 高风险 + 高影响 |
| 转移 | 交给其他人/工具处理 | 技术难题 |
| 缓解 | 采取措施降低概率/影响 | 中风险 |
| 接受 | 不处理，发生时再应对 | 低风险 |

### 风险登记册模板

```markdown
| 风险 | 概率 | 影响 | 等级 | 应对措施 | 负责人 | 状态 |
|------|------|------|------|----------|--------|------|
| 描述 | 高/中/低 | 高/中/低 | 🔴🟡🟢 | 具体措施 | 人名 | 监控中/已解决 |
```

## 🎯 与总体目标对齐

### 里程碑检查点

每个 Sprint 计划必须回答：

1. **本 Sprint 对 Demo（5/15）的贡献是什么？**
2. **本 Sprint 对 EA 上线（7/31）的贡献是什么？**
3. **如果本 Sprint 延迟，会影响哪个里程碑？**

### 对齐验证

每个 Sprint 开始前，必须验证：

```markdown
## 🎯 与总体目标对齐验证

**本 Sprint 时间：** 4/16-4/30

**对 Demo（5/15）的贡献：**
- 交付物 1 → Demo 中的哪个功能
- 交付物 2 → Demo 中的哪个功能

**对 EA 上线（7/31）的贡献：**
- 本 Sprint 是否为关键路径？是/否
- 如延迟，是否影响 EA 上线？是/否

**缓冲时间：**
- 本 Sprint 预留缓冲：X 天
- 累计缓冲剩余：X 天

**验证结论：** ✅ 对齐 / ❌ 不对齐（需调整）
```

## 🔄 变更管理

### 变更流程

```
提出变更 → 评估影响 → Kay 审批 → 更新计划 → 执行变更
```

### 变更评估模板

```markdown
## 变更请求

**变更内容：** 描述要改什么

**变更原因：** 为什么需要改

**影响评估：**
- 对 Sprint X 的影响：+X 天 / -X 天
- 对 Demo（5/15）的影响：无 / 延迟 X 天
- 对 EA 上线（7/31）的影响：无 / 延迟 X 天

**替代方案：** 有没有其他方法达到同样目的

**建议：** 批准 / 拒绝 / 修改后批准
```

### 变更纪律

- ❌ 不允许口头变更，必须书面记录
- ❌ 不允许先执行后补文档
- ✅ 所有变更必须 Kay 审批
- ✅ 变更后必须更新相关计划文件

## 📈 质量管理

### 完成定义（DoD - Definition of Done）

每个任务必须满足以下条件才算完成：

- [ ] 功能已实现
- [ ] 自测通过
- [ ] 代码已提交（如适用）
- [ ] 文档已更新（如适用）
- [ ] Kay 验收通过（P0 任务必须）

### 验收标准

每个任务必须有可量化的验收标准：

```
好的验收标准：
✅ "战斗系统可运行，包含攻击/防御/技能 3 个功能"
✅ "对话系统支持 3 层分支选择"
✅ "AI 生成 10 张立绘，风格一致性>0.85"

差的验收标准：
❌ "完成战斗系统"
❌ "优化性能"
❌ "提升质量"
```
